Vector Search · 向量检索

在高维空间中"大海捞针"——如何从百万向量中毫秒级找到最相似的几条。

余弦相似度 (Cosine Similarity)

Cosine Similarity

白话解释

余弦相似度是衡量两个向量"方向有多接近"的数学公式。想象空间中两个箭头：如果它们指向完全相同的方向，相似度为 1；如果垂直，相似度为 0；如果反向，相似度为 -1。

为什么在语义检索中用余弦相似度而非欧氏距离？因为在高维空间中，向量的长度（模长）常常受文本长度影响——一篇长文章自然产生模长更大的向量，但这不代表它语义更强。余弦相似度只看方向不看长度，因此能公平比较短查询和长文档。

// 余弦相似度公式 cos(θ) = (A · B) / (|A| × |B|) // 其中： // A·B = 点积 = a1*b1 + a2*b2 + ... + an*bn // |A| = A的模长 = sqrt(a1² + a2² + ... + an²) // JavaScript 实现 function cosineSimilarity(vecA, vecB) { let dot = 0, magA = 0, magB = 0; for (let i = 0; i < vecA.length; i++) { dot += vecA[i] * vecB[i]; magA += vecA[i] * vecA[i]; magB += vecB[i] * vecB[i]; } return dot / (Math.sqrt(magA) * Math.sqrt(magB)); // 返回值 [-1, 1]，1 = 完全相同方向 }

L2 归一化 (L2 Normalization)

L2 Normalization / 单位向量化

白话解释

L2 归一化就是把一个向量缩放到长度为 1。具体做法：把向量的每个分量除以向量的模长。归一化后，所有向量都落在单位球面上，此时余弦相似度就等于两个向量的内积——因为分母 |A|×|B| = 1×1 = 1。

这为什么有用？因为内积比余弦相似度计算更快（省去了两次开方运算），在需要做大规模检索时，这个优化可以显著提速。大多数 embedding 模型的输出默认就是 L2 归一化后的向量。

// 原始向量 A = [3, 4] → |A| = 5 // 归一化后 A' = [3/5, 4/5] = [0.6, 0.8] → |A'| = 1 // 此时 cos(A,B) = A' · B'（直接内积即可）

ANN (近似最近邻搜索)

Approximate Nearest Neighbor

白话解释

ANN 是向量检索的工程实现方案。理论上，要在 100 万个向量中找最相似的 Top-10，需要计算 100 万次余弦相似度——这在实时系统中不可接受。ANN 算法的核心思想是：牺牲一点点精度，换取数量级的速度提升。

ANN 的常见实现包括：

  • HNSW（Hierarchical Navigable Small World）：构建多层图结构，每层是上一层子集，搜索时从最稀疏的顶层开始逐层下降。类似"先看地图概览找大致区域，再放大看街道细节"。
  • IVF（Inverted File）：先聚类，把向量空间分成若干区域，搜索时只在与查询最近的那几个区域内查找。
  • PQ（Product Quantization）：把高维向量切成多段，每段单独量化，用查表替代浮点运算，大幅压缩内存和加速计算。
关键洞察：ANN 的"近似"是有理论保证的——通常能在 Top-10 结果中找回 95%+ 的真实最近邻，同时速度比暴力搜索快 100-1000 倍。在 RAG 场景中，召回率 95% 和 100% 对最终答案质量的影响微乎其微，但延迟从 500ms 降到 5ms 对用户体验是质的飞跃。

向量数据库 (Vector Database)

Vector DB · Chroma / Milvus / Pinecone

白话解释

向量数据库是专门存储和检索高维向量的数据库。传统数据库擅长"精确匹配"（找到 id=42 的记录），向量数据库擅长"相似性搜索"（找到和给定向量最接近的 10 条记录）。

主流向量数据库对比：

数据库 类型 特点 Chroma 开源·轻量 Python 原生，适合原型和中小规模 Milvus 开源·分布式 十亿级向量，云原生，性能最强 Pinecone 商业 SaaS 零运维，按量付费，开箱即用 Weaviate 开源·GraphQL 自带向量化模块，支持混合搜索 Qdrant 开源·Rust 高性能，过滤能力突出 FAISS Meta 开源 纯算法库，不是数据库，速度最快

它和传统数据库有什么区别？

传统数据库的索引是 B-Tree：数据按大小排序，适合范围查询。向量数据库的索引是图结构或聚类结构：数据按空间位置组织，适合最近邻查询。你不能用 MySQL 高效地做"找和这个向量最接近的 10 个"——这不是它设计的目标。
