RAG · 检索增强生成

给大模型装上"外部大脑"——让它在回答前先查阅你的私有知识库。

RAG (Retrieval-Augmented Generation)

检索增强生成

白话解释

RAG 是解决大模型"知识截止日期"和"幻觉"问题的关键技术。它的思路非常直观：在让模型回答问题之前，先从外部知识库中检索相关信息，把这些信息作为参考注入 prompt，然后让模型基于这些信息作答。

这就好比你在考试时——不是只靠脑子里的记忆（模型训练时学到的知识），而是可以翻书查找资料（检索外部知识库），然后基于查到的内容作答。这样既保证了答案的时效性（书是最新版），也提高了准确性（有据可查）。

为什么需要 RAG？

  • 知识截止日期：ChatGPT 的训练数据截止到某个时间点，它不知道之后发生的事。RAG 让它能访问最新文档。
  • 私有知识：你的公司内部文档、产品手册、客户数据——这些从未出现在训练数据中，但 RAG 能检索并引用。
  • 减少幻觉：有参考资料时，模型更容易给出准确答案。RAG 把"凭记忆回答"变成了"带着参考资料回答"。
  • 可溯源：RAG 可以告诉你答案来自哪个文档的哪一段——这在法律、医疗等合规场景至关重要。

RAG 四阶段 · 从文档到答案

// 阶段1: Chunk 分割 原始文档 → 按语义边界切分为段落 (256-512 tokens/chunk) 重叠 10-20% 防止边界截断导致信息丢失 // 阶段2: 向量化入库 每个 chunk → Embedding 模型 → 向量 向量 → 存入向量数据库 (Chroma / Milvus / Pinecone) // 阶段3: 相似度检索 用户查询 → Embedding 模型 → 查询向量 查询向量 → 向量数据库 ANN 搜索 → Top-K 最相似 chunks // 阶段4: 上下文增强 检索到的 chunks + 原始问题 → 拼接为增强 prompt 增强 prompt → LLM → 基于外部知识的准确回答

Chunk 分割

Document Chunking

白话解释

Chunk 分割是 RAG 的第一步——把长文档切成小块。这看起来简单，实际上是 RAG 系统质量的关键瓶颈之一。切得太小：单个 chunk 信息不完整，模型无法理解上下文；切得太大：向量中包含太多混杂信息，检索精度下降。

常见的分割策略：

  • 固定长度分割：每 512 tokens 切一刀。最简单，但会在句子中间切断。
  • 递归分割：先按段落分，段落太长再按句子分，句子还长再按词分。最常用。
  • 语义分割：用 embedding 模型判断语义转换点，在"话题变化"处分割。精度最高但计算成本高。
  • 重叠窗口：相邻 chunk 重叠 10-20%。比如 chunk1 包含 tokens 0-512，chunk2 包含 tokens 410-922。确保不会因为边界截断丢失关键信息。
关键洞察：重叠窗口是"便宜但极其有效"的技巧。它的本质是用少量存储冗余换取检索鲁棒性——如果一条关键信息恰好落在 chunk 边界上，没有重叠就会丢失，有重叠则至少被一个 chunk 完整包含。多消耗 10-20% 的存储和索引成本，换来的却是显著的召回率提升。

Rerank (重排序)

Re-ranking

白话解释

Rerank 是 RAG 流程中的一个可选的"精排"步骤。向量检索（ANN）速度快但精度有损——它可能返回 10 个结果，其中真正最相关的排在第 5 位而非第 1 位。Rerank 对 Top-K 个候选做一次更精细的相关性评分，重新排序，把最相关的结果排到最前面。

为什么不在第一步就用精细评分？因为对全部 100 万个文档做精细评分太慢了。Rerank 的策略是"粗筛 + 精排"：先用快速的向量检索缩窄到 50-100 个候选，再对这些候选做精细评分，取 Top-5 注入 prompt。

// 无 Rerank: 粗筛结果直接注入 向量检索 → 10 个结果 → 全部注入 prompt // 问题: 可能有2-3个不太相关的结果稀释上下文 // 有 Rerank: 粗筛 + 精排 向量检索 → 50 个候选 → Rerank 模型 → 取 Top-5 注入 // 优势: 上下文更紧凑、更相关，答案质量更高
