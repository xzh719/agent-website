Context Window · 上下文窗口

大模型的"短期记忆"——它能同时"看到"多少内容，以及这如何影响一切。

上下文窗口 (Context Window)

Context Window / 上下文长度

白话解释

上下文窗口是大模型在一次推理中能"同时看到"的最大文本量。它以大模型的基本单位——token——来衡量。比如 GPT-4 Turbo 的上下文窗口是 128K tokens，大约相当于一本 300 页的书。

这个窗口包括了你的所有输入——系统提示词（system prompt）、对话历史、用户当前问题——以及模型生成的所有输出。当对话持续进行，窗口逐渐被填满，最早的内容会被"遗忘"（实际上是被截断或压缩）。

为什么这很重要？

  • 对话长度限制：你和 ChatGPT 聊了太久之后，它开始"忘记"最初说的事情——这就是上下文窗口满了。
  • 文档分析能力：窗口大小决定了模型能一次性分析多长的文档。128K 窗口可以一次读完一本中篇小说。
  • Agent 记忆边界：在 Agent 系统中，上下文窗口就是短期记忆的硬上限。超过窗口的历史必须被压缩或放弃。
  • 成本：更大的窗口意味着每次推理都要处理更多 token，API 费用随窗口使用量线性增长。

主流模型上下文窗口

模型 上下文窗口 约等于 Claude 3.5 Sonnet 200K tokens ~500 页 GPT-4 Turbo 128K tokens ~300 页 Gemini 1.5 Pro 1M tokens ~2500 页 (实验性) DeepSeek V3 128K tokens ~300 页 Claude Code (实际) 200K tokens 终端会话全部上下文
关键洞察：上下文窗口不是"越大越好"。更大的窗口意味着更高的延迟和成本——每次生成下一个 token 时，模型都要重新计算窗口中所有 token 的注意力。这就是为什么 1M 窗口的 Gemini 在实际使用中可能比 128K 窗口的 GPT-4 更慢且更贵。窗口大小是"能力"和"成本"之间的工程权衡。

滑动窗口 (Sliding Window)

Sliding Window

白话解释

滑动窗口是处理"上下文溢出"的最简单策略：当对话历史超过窗口上限时，只保留最近的 N 条消息，丢弃更早的内容。就像一个只能装 10 个球的管子——塞进第 11 个球时，第一个球从另一端掉出来。

优点：实现简单，开销为零。缺点：早期的重要信息（如用户一开始设定的任务目标、关键约束条件）会被无情遗忘。这就是为什么有时候 ChatGPT 聊久了会"跑偏"。

摘要压缩 (Summarization)

Context Compression

白话解释

摘要压缩比滑动窗口更智能：当上下文接近窗口上限时，不是直接丢弃旧内容，而是用模型将旧内容总结成一段紧凑的摘要，保留关键信息、压缩冗余描述。然后用这段摘要替代原始历史，腾出空间给新的对话。

这个过程有时被称为"递归摘要"——因为它可以嵌套：当摘要本身也占满了窗口，就对摘要再做摘要。每一层压缩都在提升信息密度。

// 原始对话 (3000 tokens) 用户: 帮我分析这份财报... 助手: 好的，Q1 营收增长 15%... 用户: 那么 Q2 呢... 助手: Q2 略有下滑... ... 30 轮对话后 ... // 摘要压缩后 (200 tokens) "用户在分析公司财报。已讨论 Q1-Q4 营收数据：Q1 +15%, Q2 -3%, Q3 +8%, Q4 +22%。 关键发现：Q4 增长主要由新产品线驱动。用户要求在下周前完成完整分析报告。" // 释放了 2800 tokens 给新的对话轮次
