[ LAYER_01 ]

感知层 · Perception

sys.listen(inputs) → tokenize → embed → context_window

输入源多样化

Agent 的感官远不止文本输入。用户消息、工具返回结果、系统事件、文件内容——这些异构信号流共同构成感知层的原始输入。感知层的核心任务是将一切外部信号统一转化为模型可消费的 token 序列。

Tokenization 原理

所有输入文本首先经过分词器（如 GPT 的 BPE — Byte Pair Encoding）切分为 token。BPE 从字符级开始，统计高频相邻对并合并为一个新 token，迭代至预设词表大小（如 100K）。这解释了为什么中文通常比英文消耗更多 token——每个汉字可能被拆为 1-3 个 token。

// Tokenization 示意：BPE 合并过程 // 输入："Agent 架构解析" // Step 1: ['A','g','e','n','t',' ','架','构','解','析'] // Step 2: 高频对合并 → ['Ag','en','t',' ','架构','解','析'] // Step 3: 继续合并 → ['Agent',' ','架构','解析'] // 实际词表可达 100K+ 个 token

上下文窗口管理

每个模型有固定的上下文窗口（如 128K tokens）。当对话历史超出窗口时，需要策略性取舍：滑动窗口保留最近 N 条消息，摘要压缩将早期对话浓缩为一段摘要。感知层负责在信息完整性与 token 预算之间做动态平衡。

[ LAYER_02 ]

记忆层 · Memory

db.retrieve(vectors) → top_k → rerank → augment

三种记忆模型

借鉴认知科学，Agent 的记忆分为三个层次：短期记忆 — 当前对话的完整上下文（受限于上下文窗口）；长期记忆 — 跨会话持久化的知识（向量数据库 + RAG）；工作记忆 — 当前任务执行中的中间状态（scratchpad / 推理缓存）。

向量嵌入 — 记忆的数学基础

这是整个记忆系统最底层的秘密。一段文本经过 Embedding 模型（如 text-embedding-3-small）被映射为高维空间中的一个点——通常是一个 1536 或 3072 维的浮点数向量。语义相近的文本在这个空间中距离更近。

关键洞察：向量不是存储文本，而是编码语义。两个向量之间的余弦相似度直接反映其语义关联程度。

// 余弦相似度 — 向量检索的核心算法 function cosineSimilarity(vecA, vecB) { let dot = 0, magA = 0, magB = 0; for (let i = 0; i < vecA.length; i++) { dot += vecA[i] * vecB[i]; magA += vecA[i] * vecA[i]; magB += vecB[i] * vecB[i]; } return dot / (Math.sqrt(magA) * Math.sqrt(magB)); // 返回值 [-1, 1]，接近 1 = 高度相关 }
为什么余弦相似度比欧氏距离更适合语义检索？在高维空间中，欧氏距离受向量模长影响——长文本自然产生模长更大的向量，但这不代表语义更强。余弦相似度只关心方向，忽略模长，因此能公平比较不同长度的文本。这也是为什么 embedding 模型通常会做 L2 归一化——让所有向量落在单位球面上，此时余弦相似度等价于内积。

RAG — 检索增强生成

RAG 是长期记忆的工程实现范式，分为四个阶段：
① Chunk 分割 — 将文档按语义边界切分为段落（通常 256-512 tokens），重叠 10-20% 防止边界截断；
② 向量化入库 — 每个 chunk 经 embedding 模型转为向量，存入向量数据库（Chroma / Milvus / Pinecone）；
③ 相似度检索 — 用户查询同样被向量化，在数据库中执行 ANN（近似最近邻）搜索，返回 Top-K 个最相似的 chunk；
④ 上下文增强 — 将检索到的 chunk 作为额外上下文注入 prompt，让 LLM 基于外部知识作答。

// RAG 检索流程伪代码 async function retrieve(query, topK = 5) { // 1. 查询向量化 const qVec = await embed(query); // 2. 向量数据库检索 (ANN) const results = await vectorDB.search(qVec, topK); // 3. 可选：重排序 (Rerank) 提升精度 const reranked = await reranker.rerank(query, results); // 4. 拼接上下文注入 prompt const context = reranked.map(r => r.text).join('\\n---\\n'); return `基于以下知识回答:\\n${context}\\n\\n问题: ${query}`; }

记忆压缩 — 对抗遗忘

当短期记忆超出上下文窗口时，不能简单截断。工程实践包括：递归摘要（将旧对话折叠为总结段落）、滑动窗口 + 摘要（窗口内保留原文，窗口外仅存摘要）、实体记忆（提取关键实体和决策，结构化存储）。压缩的核心是信息密度的提升——用更少的 token 承载更多的语义。

推理层 · Brain

llm.reason(CoT) · sys.reasoning_mode = DECOUPLED

剥离一切框架表象，LLM 的本质仅仅是概率预测。要将这个预测机变成智能体，必须强制它进行思维链（CoT）推演——将中间推理与最终输出解耦，让每一步思考都可见、可审计、可纠错。

[ BRAIN :: CoT ]

思维链深度解析

逐步推理 · 自洽性验证 · 思维树搜索

CoT 为什么有效？

LLM 是自回归模型——每个 token 的生成都依赖于之前所有 token。CoT 通过在前文写入推理步骤，扩大了有效上下文：模型在生成答案时不仅能看问题，还能看到自己写下的分析过程。这本质是将隐式计算外化为显式文本，让模型自我引导。

思维树 (Tree of Thoughts)

进阶玩法：不只一条推理路径，而是同时展开多条，在每一步做剪枝评估。BFS 广度搜索 + LLM 评分器，将推理建模为启发式搜索问题。当问题的分支因子高（如数学证明、编程方案选型）时，ToT 显著优于单链 CoT。

CoT / ToT 的本质不是让模型"更聪明"，而是扩大有效上下文。LLM 是自回归模型——每个 token 的生成概率依赖于前文所有 token。当你在 prompt 中写下推理步骤，这些步骤成为模型生成最终答案时的"锚点"，显著降低偏离正确路径的概率。这也是为什么零样本 CoT 只需加一句"Let's think step by step"就能提升准确率。
// CoT 提示模板 const cotPrompt = ` 请逐步推理，每一步标明 [步骤N]： [步骤1] 分析已知条件和约束 [步骤2] 提出可能的解决路径 [步骤3] 逐一评估每条路径的可行性 [步骤4] 选择最优路径并给出最终答案 问题: ${userQuery}`;
const thought = await llm.reason(query, { mode: 'chain-of-thought', steps: 4 }); // CoT → 外化推理

[ LOOP :: ReAct ]

ReAct 循环机制

while(task_not_done) { think → act → observe }

循环三阶段

Reason — LLM 分析当前状态、历史观察、可用工具，生成下一步推理和行动计划。
Act — 将计划转化为具体的工具调用（function call），传入参数，等待执行结果。
Observe — 接收工具返回结果，判断任务是否完成。未完成则带着新观察回到 Reason 阶段。

// ReAct 主循环 async function reactLoop(task, tools, maxSteps = 10) { let history = [{ role: 'user', content: task }]; for (let step = 0; step < maxSteps; step++) { const response = await llm.chat(history, tools); if (response.finish_reason === 'stop') break; const result = await executeTool(response.tool_call); history.push(response.message); history.push({ role: 'tool', content: result }); } return history; }

执行层 · Action

tool.execute(json) → validate → sandbox → observe

Function Calling 协议

当 LLM 决定调用工具时，它不直接执行——而是输出一个结构化的 JSON 函数调用（函数名 + 参数）。Agent 框架拦截这个输出，校验参数合法性，在沙箱中真正执行，再将结果反馈给 LLM。LLM 本身是纯文本引擎，它不跑代码——它只是提议要跑什么代码。

这里有一个容易被忽视的安全边界：LLM 的输出是不可信的。它可能幻觉出不存在的函数名、不合理的参数值、甚至尝试注入恶意命令。框架必须在 LLM 和真实执行环境之间充当"防火墙"——校验 schema、白名单工具名、限制执行权限。永远不要让 LLM 的原始输出直接接触系统调用。

参数校验与 Schema

每个工具通过 JSON Schema 声明其接口契约：参数名、类型（string/number/boolean/object）、是否必填、枚举范围。框架在调用前做 schema validation，拦截 LLM 可能产生的幻觉参数（如文件路径不存在、数值越界）。这层校验是 Agent 安全性的第一道防线。

// 工具定义 JSON Schema { "name": "read_file", "description": "读取指定路径的文件内容", "parameters": { "type": "object", "properties": { "path": { "type": "string", "description": "文件绝对路径" }, "encoding": { "type": "string", "enum": ["utf-8", "base64"] } }, "required": ["path"] } }

沙箱与容错

执行层必须假设工具调用 会失败。文件不存在、API 超时、权限不足——这些异常不能使 Agent 崩溃。工程实践包括：超时控制（每个工具调用有独立 timeout）、重试策略（指数退避）、错误注入 LLM（将异常信息作为 observation 返还，让 Agent 自行调整策略）。

const result = await tool.execute({ name: 'api_call', params: validated }); // JSON Schema → validate → sandbox

1. 输入接收与 Tokenization
用户文本进入系统，BPE Tokenizer 将中文切分为 token 序列。同时系统注入本次对话的元数据——用户 ID、位置权限、当前时间戳，一并打包为上下文字段。
